### Transaction Data Structure

Transactions are stored in the `transactions` Firestore collection:

```javascript
{
  transaction_id: string,
  transaction_type: string, // 'sale', 'refund', 'adjustment', 'payout', 'fee'
  
  // Amounts
  amount: number,
  currency: string,
  platform_fee: number,
  service_fee: number,
  net_amount: number,
  
  // Associations
  event_id: string,
  event_name: string,
  order_id: string,
  ticket_ids: [string],
  client_id: string,
  customer_id: string,
  
  // Payment details
  payment_method: string,
  payment_provider: string,
  payment_reference: string,
  
  // Status
  status: string, // 'pending', 'completed', 'failed', 'reversed'
  
  // Timestamps
  date: {
    created: timestamp,
    completed: timestamp,
    updated: timestamp
  },
  
  // Notes
  description: string,
  notes: string,
  processed_by: string
}
```

### Transaction Types

<Accordion title="Sale Transactions">
  Created when tickets are purchased:
  - Amount: Total ticket price including fees
  - Platform fee: Calculated based on configuration
  - Service fee: Per-ticket service charge
  - Net amount: Amount payable to client after fees
  - Status: Pending until payment confirmed
</Accordion>

<Accordion title="Refund Transactions">
  Created when tickets are refunded:
  - Amount: Negative value of original sale
  - References original transaction ID
  - Updates original ticket status
  - May include refund processing fee
  - Adjusts client payout amounts
</Accordion>

<Accordion title="Adjustment Transactions">
  Manual financial adjustments:
  - Administrative corrections
  - Fee waivers or additions
  - Promotional credits
  - Error corrections
  - Requires admin approval
</Accordion>

<Accordion title="Payout Transactions">
  When funds are distributed to clients:
  - Amount: Total payout to client
  - References custody account
  - Links multiple sale transactions
  - Includes payout method details
  - Records completion confirmation
</Accordion>

### Payout Processing

Payouts are processed periodically (typically weekly or monthly):

1. **Calculation**: Aggregate all completed transactions for client
2. **Fee Deduction**: Subtract platform and service fees
3. **Split Application**: Apply any revenue splits to collaborators
4. **Payout Creation**: Create payout transaction
5. **Transfer**: Initiate bank transfer to client account
6. **Confirmation**: Update status when transfer completes
7. **Notification**: Notify client of payout completion

## Financial Splits

Revenue sharing between parties:

```javascript
// From event setup (src/components/apps/events/setup-event/SetupEventSplitForm.js)
{
  splits: [{
    party_type: string, // 'platform', 'client', 'collaborator', 'venue'
    party_id: string,
    party_name: string,
    percentage: number,
    fixed_amount: number,
    split_type: 'percentage' | 'fixed'
  }],
  
  // Validation
  total_percentage: number, // Must equal 100
  
  // Application
  applied_to: 'gross' | 'net', // Before or after platform fees
}
```

### Split Calculation Example

```javascript
// Ticket sale: $100
// Platform fee: 10% = $10
// Gross: $100
// Net after platform fee: $90

// Splits applied to net:
// Client: 70% of $90 = $63
// Collaborator: 20% of $90 = $18  
// Venue: 10% of $90 = $9
```

## Reconciliation

Regular reconciliation process:

1. **Transaction Verification**: Verify all transactions are recorded
2. **Balance Checks**: Ensure custody account balances match records
3. **Payout Verification**: Confirm all payouts were received by clients
4. **Fee Calculation**: Verify platform fees are correctly calculated
5. **Split Validation**: Ensure revenue splits are applied correctly
6. **Discrepancy Resolution**: Investigate and resolve any mismatches
